#

针对微信浏览器中的页面，如需使用微信客户端相关的功能，需使用其jsSDK, 常见操作如分享，图片，声音，以及对原生菜单选项的操作等
需引入微信JSSDK

开发流程及注意

  1. 微信公众号的申请，及其域名绑定
  2. 微信JS-SDK说明文档
  3. access_token , jsapi_tiket的获取，必须缓存，最长两个小时
  4. 由于url中会带参数，别人分享的微信链接打开时会附带上from=singlemessage&isappinstalled=0参数，故，为能正确提取url参数，必须全字段进行encodeURIComponent
  5. 请求的当前页面中，若有中文，则必须保持页面的URL与参数中的完全一致(不包含hash)
  6. 获取微信conf可走异步接口获取，在回调函数中再执行相关的初始化
  7. 在调用checkjsapi，chooseImage接口时提示，errmsg：function not exist;是因为版本没有升级
  8. 在版本没有升级到6.0.2,分享到朋友圈，分享给好友的接口可用
  9. wx相关api均需在 wx.ready 后回调进行
  10. 在线demo

分享相关API

  1. event包括 title link imgUrl desc trigger(function) complete(function)等内容
  2. jsJDK中的分享，只是将数据绑定到相关的按钮上，可多次绑定(覆盖), 如果单页面中，event中仅内容可变，其它均相同，为防止多次绑定该时间，可通过更改 event.triger 中更新数据

图片相关API

  1. wx.chooseImage 选择图后，wx.uploadImage进行上传时，需保证每次只上传一张图，，应该在每次wx.uploadImage成功后的回调中再次上传下一张图片，以保证每次只传一张图，上传图片
  2. 传图成功后，应记下每张图的serveceID
  3. 用此ID将图从微信服务器转存到自己的服务器，微信服务器上只能临时保存3天

一. 简介

GitBlog是一个简单易用的Markdown博客系统，它不需要数据库，没有管理后台功能，更新博客只需要添加你写好的Markdown文件即可。它摆脱了在线编辑器排版困难，无法实时预览的缺点，一切都交给Markdown来完成，一篇博客就是一个Markdown文件。同时也支持评论，代码高亮，数学公式，页面PV统计等常用功能。GitBlog提供了不同的主题样式，你可以根据自己的喜好配置，如果你想自己制作博客主题，也是非常容易的。GitBlog还支持整站静态导出，你完全可以导出整站静态网页部署到Github Pages。

二. 功能特点

  1. 使用Markdown
  2. 评论框
  3. 代码高亮
  4. PV统计
  5. Latex数学公式
  6. 自制主题
  7. 响应式
  8. 全站静态导出
  9. 良好的SEO

三. GitBlog优势

  1. 无需数据库，系统更轻量，移植更方便
  2. 使用Markdown编写，摆脱后台编辑排版困难，无法实时预览的缺点
  3. 可全站静态导出
  4. 配置灵活，可自由开关某些功能
  5. 多主题支持，可自制主题
  6. 博客，分类，标签，归档

四. 环境要求

PHP 5.2.4+

五. 安装步骤

  1. 下载GitBlog源代码
  2. 解压上传到你的PHP网站根目录
  3. 打开浏览器，访问网站首页
  4. 上传Markdown文件到posts文件夹

代码组织结构

在相应的html中通过 script data-main=”js/page/app.js” src=”js/require.js” 引入代码.
require.js是所有页面js的入口， data-main对应的才是该页面真实的js入口
js目录结构中，page中为每个页面对应的入口，每个页面相当于是一个应用，这些入口js是应用级的，不可复用
module为针对每个功能开发的模块，requirejs的模板就是鼓励代码模块化
lib为按需载入的各个第三方库，如jqurey,等
原则：尽量模块化，高复用，全局变量要尽量少

page 入口文件

requirejs config

config详解

  • 所有模块，依赖都默认为是js,无需加上.js后缀
  • config需在入口处，保证先执行，也可写在html中
  • paths 映射不直接放置在baseUrl下的模块名，一般相对路径，之后，可用映射名代替这一模块，一般用以映射lib库, 映射对象也可是绝对路径，或者完整URL(with https?)
  • shim 可将非AMD规范的依赖规范化，之后，每项可直接作为依赖直接使用了, 。每项包含3个参数：
  1. exports可将这一依赖注入到全局中，作为全局变量
  2. deps:[该依赖的依赖列表]，其依赖在配置时并未加载，只有当改依赖需要被载入时
  3. init: construct function + 解决变量冲突的问题(由于不符合AMD规范的库可能使用相同的全局变量，从而引起冲突，一般是通过第三方依赖的noConflict方法实现)
  • map 用以进行版本映射，以在不同页面中使用不同的依赖版本，一般用不着，谁没事弄几个版本的代码一起用啊
  • config 应用级的参数传递，用以向模块(modules中的文件)传递参数，应用级，即不同的页面调用相同的子模块实现不同的效果，向子模块传递参数，每一项是一个模块，以及该模块需使用的参数,如，在 模块getMore中，可使用module.config().total获取该参数
  • 其他，packages(引入包，其住模块为index.js)，urlArgs, scriptType…
